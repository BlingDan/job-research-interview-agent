# Agent-Pilot 下一阶段计划：先把桌面端做成真实可对话的飞书协同入口

## Summary
当前最大缺口不是 Android，也不是更深的架构整理，而是：**桌面端还不是一个可正常对话的真实产品端**。  
下一阶段应先把 Windows 端从“任务 companion 面板”升级为“聊天 + 任务双栏”的桌面协同端，并且消息来源优先**强绑定真实飞书消息链路**，而不是继续只靠本地命令模拟。

目标是满足赛题里这几个核心点：
- `IM` 仍然是主入口
- 桌面端能像真实办公客户端一样连续对话
- Agent 在对话里驱动任务，而不是只有一个命令输入框
- 任务、产物、状态在聊天和任务视图中联动展示

## Key Changes
### 1. Windows 端升级为“聊天 + 任务双栏”
- 将当前 [windows_app.dart](E:\codespace\job-research-interview-agent\clients\agent_pilot_flutter\lib\surfaces\windows\windows_app.dart) 的主界面重构为双栏：
  - 左栏：IM 对话流
  - 右栏：当前任务详情、步骤、产物、待处理动作
- 左栏至少包含：
  - 消息列表
  - 输入框
  - Agent 回复
  - 系统状态消息（规划中、等待确认、生成中、已完成）
- 右栏继续复用现有任务详情和动作组件，但默认跟随当前活跃会话 / 当前活跃任务切换。
- 当前“Inject IM Command”这种调试式命名和单输入框入口应退场，不再作为桌面端主体验。

### 2. 消息来源优先绑定真实飞书链路
- 桌面端下一阶段不再以“本地伪 IM”作为主目标，而是优先消费真实飞书消息链路。
- 实现上保持后端仍然是统一中枢，但桌面端展示的数据应围绕：
  - 当前飞书会话消息
  - Agent 对同一会话的回复
  - 从该会话触发的任务状态
- `scripts/lark_event_listener.py` 和现有 Feishu/Lark integrations 继续作为真实 IM 驱动链路的基础。
- 允许本地调试保留 fallback，但比赛主演示路径必须围绕真实飞书消息来讲，不再把桌面端描述成“命令面板”。

### 3. 补桌面端聊天数据模型与接口视图
- 当前公开任务 API 继续保留不动：
  - `/api/im/*`
  - `/api/assistant/*`
  - `/api/windows/*`
- 但桌面端需要新增或补齐“会话视图”层面的读取能力，至少支持：
  - 当前会话消息列表
  - 当前会话关联任务摘要
  - 当前活跃任务详情
- 如果后端缺少聊天流读取接口，本轮优先补“桌面端聊天视图所需的最小会话查询接口”，而不是另起一套桌面端专用任务协议。
- 任务仍然以统一 `task_id` 为中心，聊天只作为会话和上下文入口，不改共享状态主模型。

### 4. 重新排序后续优先级
- 第一优先级：
  - Windows 真实对话桌面端
- 第二优先级：
  - 基于同一任务模型补 Android
- 第三优先级：
  - 深化 Doc / Slides / Canvas 的真实飞书套件联动
- 也就是说，Android 不是取消，而是延后到桌面端“正常对话”成立之后再补，这样更符合当前真实缺口。

### 5. 演示脚本同步改写
- 比赛 demo 主路径调整为：
  1. 在飞书真实会话中发起任务
  2. Windows 端左栏看到真实对话流和 Agent 回复
  3. Windows 端右栏同步出现任务计划、步骤、产物、待确认动作
  4. 在桌面端完成确认或修订
  5. 同一任务在后续移动端再接续
- 这样桌面端承担“主演示端”，Android 承担“跨端延续证明端”，叙事会更顺。

## Public Interfaces / Types
- 现有任务接口保持不变：
  - `POST /api/im/commands`
  - `POST /api/im/events`
  - `GET /api/assistant/tasks/{task_id}`
  - `POST /api/assistant/tasks/{task_id}/actions/*`
  - `GET /api/windows/*`
- 本轮允许补充最小聊天视图接口，但要求：
  - 不改变现有任务接口语义
  - 不引入桌面端专属任务模型
  - 会话消息与任务状态通过共享标识关联
- Windows 前端最终应同时消费：
  - 会话消息视图
  - 统一任务快照/详情视图

## Test Plan
- 桌面端：
  - Windows 应用可打开双栏主界面
  - 左栏可展示会话消息与 Agent 回复
  - 输入一条自然语言后，能触发真实飞书消息链路或其等价调试链路
  - 右栏能联动展示当前任务详情、步骤、产物、动作
- 后端：
  - IM 入口继续能创建任务
  - 任务状态仍通过统一 `task_id` 推进
  - 若新增聊天读取接口，应覆盖成功路径和无消息/无任务路径
- 联调：
  - 飞书会话消息 -> Agent 回复 -> Windows 聊天流显示
  - Windows 上执行 `confirm` / `revise` 后，任务详情刷新一致
  - cockpit 继续能看到同一任务状态变化
- 回归：
  - 现有 `tests/test_agent_pilot_api.py`
  - 现有 `tests/test_windows_surface_contract.py`
  - 新增桌面聊天流契约测试或最小集成测试

## Assumptions
- 当前 Windows 已能运行，但产品形态仍然不达标；本轮核心是“体验升级为真实对话端”，不是桌面环境问题排查。
- 桌面端将作为下一阶段主演示端，Android 延后到桌面端成型后再补。
- 消息来源优先绑定真实飞书链路；本地模拟只作为调试后备，不作为比赛主叙事。
- `cockpit` 保持观察和操盘定位，不承担桌面端聊天主入口职责。
