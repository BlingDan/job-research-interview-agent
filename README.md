# 基于 IM 的办公协同智能助手

面向飞书校园 AI 挑战赛的校招协同智能助手方案，聚焦“IM 发起任务、文档沉淀过程、演示稿输出成果”的一体化效率提升。项目以校招求职准备为核心场景，将岗位研究、公司调研、面试准备和多人协作整合为可编排的智能工作流。

## 项目概述

校招准备通常分散在多个环节之中：岗位 JD 在群聊中流转，公司资料需要临时检索，面试重点依赖个人整理，最终还要输出文档、提纲或展示材料。这一过程跨越即时沟通、资料沉淀和成果表达多个工具，存在信息割裂、重复整理、协作效率低等典型问题。

本项目围绕这一痛点，构建一个嵌入飞书协同链路的智能助手：

- 以 IM 消息作为任务入口，接收岗位 JD、公司名称、面试主题和补充资料
- 通过 Agent 自动完成任务拆解、公开信息搜索、本地资料检索和阶段性总结
- 将研究过程沉淀为结构化文档，支持协作查看、批注和持续迭代
- 在文档基础上进一步生成面试准备提纲或演示稿内容，形成从沟通到成果的闭环

项目定位不是单轮问答工具，而是面向真实办公协同场景的任务型智能助手。

## 赛题契合度

项目与“基于 IM 的办公协同智能助手”方向高度一致，主要体现在以下几个方面：

- **IM 驱动任务发起**：用户可在飞书单聊或群聊中直接提交 JD、公司信息、问题主题和附件资料。
- **文档承载过程协同**：系统将中间研究结果与最终结论持续沉淀到结构化文档中，支持多人查看、批注与版本迭代。
- **演示稿扩展成果表达**：研究输出可进一步组织为面试准备提纲、项目陈述材料或展示页内容，覆盖从理解到表达的完整流程。
- **模块化场景编排**：现有 `Planner / Search / RAG / Memory / Report / SSE` 架构可直接映射为模块化流程编排能力。
- **多端协同能力明确**：IM 发起、文档协作、演示输出天然适合移动端与桌面端联动。

## 典型应用场景

### 场景一：群内发起岗位研究

用户在飞书群中发送岗位 JD 或职位链接后，系统自动识别任务并拆分研究步骤，包括岗位能力要求、公司业务背景、面试高频考点和候选人准备重点。执行过程中，系统通过消息流实时反馈进度，并将阶段成果同步沉淀到研究文档中。

### 场景二：导师与同学协作完善面试方案

研究文档生成后，导师、学长学姐或同学可在飞书文档中继续补充意见，例如岗位理解是否准确、项目经历如何映射、追问清单是否充分。系统基于协作反馈继续迭代内容，形成持续优化的面试准备方案。

### 场景三：从研究结果延展到汇报材料

当候选人需要准备自我介绍、项目陈述或复盘汇报时，系统可将结构化研究结果进一步组织为演示稿大纲或展示内容，缩短从“完成准备”到“完成表达”的转换路径。

## 核心价值

- **流程整合**：将 IM 沟通、信息检索、文档沉淀和成果输出统一到同一条智能工作流中。
- **效率提升**：减少手工收集、重复整理和多工具来回切换带来的时间损耗。
- **协作增强**：研究结果可沉淀、可共享、可批注、可持续迭代，适合真实团队协同。
- **场景真实**：校招求职准备具有高频、刚需、强协作的特点，易于体现产品落地价值。
- **能力可复用**：相同的工作流框架可扩展到实习申请、行业调研、课程项目协同等更多办公场景。

## 方案架构

项目采用“输入理解 -> 任务规划 -> 研究执行 -> 内容汇总 -> 协作沉淀 -> 成果输出”的链路设计。

## 当前原型能力

当前仓库处于基础原型阶段，已完成核心工作流的初步打通，并具备以下能力：

### 结构化任务输入

系统支持接收以下输入字段：

- `jd_text`：岗位 JD 文本
- `company_name`：目标公司名称
- `interview_topic`：面试主题
- `local_context_path`：本地资料路径
- `user_note`：用户补充说明

### 研究流程骨架

原型已具备完整的研究流程模块：

- `Planner`：任务拆解与研究规划
- `Search`：围绕岗位、公司与面试主题的公开搜索
- `RAG`：本地资料检索扩展接口
- `Memory`：上下文记忆与持续迭代能力预留
- `Report`：结构化研究报告生成
- `SSE`：阶段进度的流式回传

### 基础服务接口

当前 FastAPI 原型已包含：

- 健康检查接口
- 任务创建接口
- 流式进度接口

### 过程产物落盘

系统会将规划结果、搜索结果、研究上下文、阶段摘要和最终报告写入 `workspace/tasks/<task_id>/`，为后续接入飞书文档、协作回写和版本追踪提供基础。

## 飞书协同形态设计

### IM 任务入口

- 在单聊或群聊中提交岗位 JD、公司信息和问题主题
- 支持上传附件或引用已有材料
- 在会话中实时回传当前进度、关键发现与任务状态

### 文档协同中台

- 自动创建与任务关联的研究文档
- 按岗位分析、公司分析、面试追问、准备建议等结构生成内容
- 支持导师、同学、学长学姐继续批注和共创

### 演示稿成果输出

- 从研究文档自动提炼页面结构和核心表达点
- 生成适用于面试准备、项目介绍和复盘汇报的展示内容
- 打通从资料分析到表达呈现的最后一段流程

### 模块化工作流编排

基于现有架构，后续可配置出不同协同模板，例如：

- 校招岗位研究
- 高频面试追问准备
- 公司背景速读
- 导师协同点评
- 复盘汇报生成

## 项目优势

### 真实业务场景清晰

项目直接面向校招求职这一高频刚需场景，问题清晰、用户明确、价值闭环完整，适合在比赛中展示真实落地能力。

### 从对话到成果的全链路闭环

系统不止提供信息回答，而是覆盖任务发起、研究执行、过程沉淀与成果输出，形成完整的协同效率提升链路。

### 协作属性突出

项目强调的是“AI 参与协同流程”，而不是“AI 替代所有流程”。这一定位更符合飞书生态中对组织协作与共创效率的关注。

### 技术结构与产品表达一致

现有 `Planner / Summarizer / Report` 分层设计既能支撑后续工程演进，也便于在项目展示中清晰说明模块边界与编排逻辑。

## 项目结构

当前核心目录如下：

```text
app/
├─ agents/      # Agent prompt 与模型调用封装
├─ api/         # FastAPI 路由
├─ core/        # 配置与 LLM 客户端
├─ schemas/     # 输入输出与状态结构定义
├─ services/    # 编排、搜索、RAG、记忆等服务层
├─ tools/       # 搜索与检索相关工具层
└─ main.py      # 应用入口

tests/          # 当前原型测试
docs/           # 开发记录与实现说明
workspace/      # 研究任务运行产物
```

## 本地运行

根据仓库约定，默认优先使用 `uv`。

### 启动服务

```bash
uv run uvicorn app.main:app --reload
```

### 运行测试

```bash
uv run pytest
```
